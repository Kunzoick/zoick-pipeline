# ADR-003 — Idempotency Approach

**Status:** Accepted
**Project:** zoick-pipeline — Async Incident Processing Pipeline
---

## Context
RabbitMQ guarantees at-least-once delivery. This means the same message
can arrive at the consumer more than once — on broker restart, network
partition, or unacknowledged delivery timeout. Without an idempotency
mechanism, an incident could be processed multiple times, causing
duplicate state transitions, incorrect retry counts, or corrupted records.

An idempotency mechanism must satisfy two requirements:
1. The same message delivered twice produces the same result as delivered once
2. Duplicate detection must be durable — surviving application restarts
---

## Decision
Database-backed idempotency using `eventId` as the deduplication key.
- Every `IncidentCreatedEvent` carries a unique `eventId` generated by
  the Trust API publisher at the moment of outbox row creation
- When the consumer receives an event, `ProcessingService` calls
  `processingRepository.findByEventId(eventId)` before any processing begins
- If a `COMPLETED` or `DEAD` record exists for that `eventId` — acknowledge
  and return immediately, no processing occurs
- If no record exists — create a `PENDING` record and proceed
- A `UNIQUE` constraint on `processing_records.event_id` provides a
  database-level safety net against race conditions
---

## Alternatives Considered
### Redis-based deduplication cache
A Redis set of processed `eventId` values with a TTL was considered.
Rejected for two reasons:

First, Redis is volatile. If the cache is evicted or Redis restarts,
the deduplication history is lost. A message that was processed before
the eviction would be processed again — silently, with no error. The
database never loses committed records.

Second, the Trust API already uses Redis for rate limiting with a
fail-open strategy. Adding Redis to the critical path of idempotency
would mean a Redis failure could cause duplicate processing — a much
worse failure mode than the rate limit fail-open.

### Using `incidentId` as the deduplication key

`incidentId` was considered instead of `eventId`. Rejected because the
two identifiers serve different purposes.

`incidentId` is the business identifier — it identifies the incident as
a domain object. `eventId` is the message identifier — it identifies this
specific delivery of this specific event. The same incident should never
produce two events, but if the outbox poller publishes the same row twice
(at-least-once delivery from the poller), two messages with the same
`incidentId` but different `eventId` values would arrive. Using `incidentId`
as the key would incorrectly treat the second delivery as a duplicate even
if the first was never processed. Using `eventId` correctly identifies
the specific message.
---

## Consequences
**Positive:**
- Durable deduplication — survives application and database restarts
- `UNIQUE` constraint enforces correctness at the database level even
  if application-level check is bypassed by a race condition
- Deduplication key is the message identifier, not the business identifier —
  correct semantics
- Full processing history visible in `processing_records` table

**Negative:**
- Every message requires a database read before processing begins
- At high volume this creates database pressure — mitigated at scale
  by indexing `event_id` and using a read replica for idempotency checks

**Dependency note:** The outbox poller in the Trust API is intentionally
at-least-once. If a poller cycle crashes between publish and marking
the outbox row `PUBLISHED`, the same row is republished on the next cycle.
Idempotency on the consumer is not optional — it is load-bearing. Without
it, duplicate processing is guaranteed under normal failure conditions,
not just edge cases.
